 18  # Strongly Connected Components
 19  
 20  This file defines tactics to construct proofs of equivalences between a set of mutually equivalent
 21  propositions. The tactics use implications transitively to find sets of equivalent propositions.
 22  
 23  ## Implementation notes
 24  
 25  The tactics use a strongly connected components algorithm on a graph where propositions are
 26  vertices and edges are proofs that the source implies the target. The strongly connected components
 27  are therefore sets of propositions that are pairwise equivalent to each other.
 28  
 29  The resulting strongly connected components are encoded in a disjoint set data structure to
 30  facilitate the construction of equivalence proofs between two arbitrary members of an equivalence
 31  class.
 32  
 33  ## Possible generalizations
 34  
 35  Instead of reasoning about implications and equivalence, we could generalize the machinery to
 36  reason about arbitrary partial orders.
 37

 52  /--
 53  `closure` implements a disjoint set data structure using path compression
 54  optimization. For the sake of the scc algorithm, it also stores the preorder
 55  numbering of the equivalence graph of the local assumptions.
 56  
 57  The `expr_map` encodes a directed forest by storing for every non-root
 58  node, a reference to its parent and a proof of equivalence between
 59  that node's expression and its parent's expression. Given that data
 60  structure, checking that two nodes belong to the same tree is easy and
 61  fast by repeatedly following the parent references until a root is reached.
 62  If both nodes have the same root, they belong to the same tree, i.e. their
 63  expressions are equivalent. The proof of equivalence can be formed by
 64  composing the proofs along the edges of the paths to the root.
 65  
 66  More concretely, if we ignore preorder numbering, the set
 67  `{ {e₀,e₁,e₂,e₃}, {e₄,e₅} }` is represented as:
 68  
 69  ```
 70  e₀ → ⊥      -- no parent, i.e. e₀ is a root
 71  e₁ → e₀, p₁ -- with p₁ : e₁ ↔ e₀
 72  e₂ → e₁, p₂ -- with p₂ : e₂ ↔ e₁
 73  e₃ → e₀, p₃ -- with p₃ : e₃ ↔ e₀
 74  e₄ → ⊥      -- no parent, i.e. e₄ is a root
 75  e₅ → e₄, p₅ -- with p₅ : e₅ ↔ e₄
 76  ```
 77  
 78  We can check that `e₂` and `e₃` are equivalent by seeking the root of
 79  the tree of each. The parent of `e₂` is `e₁`, the parent of `e₁` is
 80  `e₀` and `e₀` does not have a parent, and thus, this is the root of its tree.
 81  The parent of `e₃` is `e₀` and it's also the root, the same as for `e₂` and
 82  they are therefore equivalent. We can build a proof of that equivalence by using
 83  transitivity on `p₂`, `p₁` and `p₃.symm` in that order.
 84  
 85  Similarly, we can discover that `e₂` and `e₅` aren't equivalent.
 86  
 87  A description of the path compression optimization can be found at:
 88  <https://en.wikipedia.org/wiki/Disjoint-set_data_structure#Path_compression>
