 10  # Transport multiplicative to additive
 11  
 12  This file defines an attribute `to_additive` that can be used to
 13  automatically transport theorems and definitions (but not inductive
 14  types and structures) from multiplicative theory to additive theory.
 15  
 16  To use this attribute, just write
 17  
 18  ```
 19  @[to_additive]
 20  theorem mul_comm' {α} [comm_semigroup α] (x y : α) : x * y = y * x := comm_semigroup.mul_comm
 21  ```
 22  
 23  This code will generate a theorem named `add_comm'`.  It is also
 24  possible to manually specify the name of the new declaration, and
 25  provide a documentation string.
 26  
 27  The transport tries to do the right thing in most cases using several
 28  heuristics described below.  However, in some cases it fails, and
 29  requires manual intervention.
 30  
 31  ## Implementation notes
 32  ### Handling of hidden definitions
 33  
 34  Before transporting the “main” declaration `src`, `to_additive` first
 35  scans its type and value for names starting with `src`, and transports
 36  them. This includes auxiliary definitions like `src._match_1`,
 37  `src._proof_1`.
 38  
 39  After transporting the “main” declaration, `to_additive` transports
 40  its equational lemmas.
 41  
 42  ### Structure fields and constructors
 43  
 44  If `src` is a structure, then `to_additive` automatically adds
 45  structure fields to its mapping, and similarly for constructors of
 46  inductive types.
 47  
 48  For new structures this means that `to_additive` automatically handles
 49  coercions, and for old structures it does the same, if ancestry
 50  information is present in `@[ancestor]` attributes.
 51  
 52  ### Name generation
 53  
 54  * If `@[to_additive]` is called without a `name` argument, then the
 55    new name is autogenerated.  First, it takes the longest prefix of
 56    the source name that is already known to `to_additive`, and replaces
 57    this prefix with its additive counterpart. Second, it takes the last
 58    part of the name (i.e., after the last dot), and replaces common
 59    name parts (“mul”, “one”, “inv”, “prod”) with their additive versions.
 60  
 61  * If `@[to_additive]` is called with a `name` argument `new_name`
 62    /without a dot/, then `to_additive` updates the prefix as described
 63    above, then replaces the last part of the name with `new_name`.
 64  
 65  * If `@[to_additive]` is called with a `name` argument
 66    `new_namespace.new_name` /with a dot/, then `to_additive` uses this
 67    new name as is.
 68  
 69  As a safety check, in the first two cases `to_additive` double checks
 70  that the new name differs from the original one.
 71  
 72  ### Missing features
 73  
 74  * Automatically transport structures and other inductive types.
 75  
 76  * Handle `protected` attribute. Currently all new definitions are public.
 77  
 78  * For structures, automatically generate theorems like `group α ↔
 79    add_group (additive α)`.
 80  
 81  * Mapping of prefixes that do not correspond to any definition, see
 82    `quotient_group`.
 83  
 84  * Rewrite rules for the last part of the name that work in more
 85    cases. E.g., we can replace `monoid` with `add_monoid` etc.
 86  -/
