  8  # Library notes
  9  
 10  At various places in mathlib, we leave implementation notes that are referenced from many other
 11  files. To keep track of these notes, we use the command `library_note`. This makes it easy to
 12  retrieve a list of all notes, e.g. for documentation output.
 13  
 14  An example from `meta.expr`:
 15  
 16  ```
 17  library_note "open expressions"
 18  "Some declarations work with open expressions, i.e. an expr that has free variables.
 19  Terms will free variables are not well-typed, and one should not use them in tactics like
 20  `infer_type` or `unify`. You can still do syntactic analysis/manipulation on them.
 21  The reason for working with open types is for performance: instantiating variables requires
 22  iterating through the expression. In one performance test `pi_binders` was more than 6x
 23  quicker than `mk_local_pis` (when applied to the type of all imported declarations 100x)."
 24  ```
 25  
 26  These notes can be referenced in mathlib with the syntax `Note [note id]`.
 27  Often, these references will be made in code comments (`--`) that won't be displayed in docs.
 28  If such a reference is made in a doc string or module doc, it will be linked to the corresponding
 29  note in the doc display.
 30  
 31  Since these notes are used in files imported by `tactic.core`, this file has no imports.
 32  
 33  ## Implementation details
 34  
 35  `library_note note_id note_msg` creates a declaration `` `library_note.i `` for some `i`.
 36  This declaration is a pair of strings `note_id` and `note_msg`, and it gets tagged with the
 37  `library_note` attribute.
